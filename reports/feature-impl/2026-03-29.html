Diplomacy Tick Integration

Feature Implementation Report — 29 March 2026

a) Feature Summary

What was implemented: The diplomacy tick (Step 8 of the game loop) was a complete stub that returned state unchanged. This feature replaces it with a fully functional diplomacy processing step that integrates with the existing diplomacy engine, AI decision system, and economy.

PROJECT.md item: Milestone 6 (Diplomacy & Espionage) — specifically the diplomatic stance system, treaty types, trade route income, and AI diplomacy behaviour. Also unblocks Milestone 7 (AI & Single Player) AI diplomatic decisions.

Why it matters: Without this feature, diplomacy was entirely static. Attitudes never decayed, treaties never expired, trade income from diplomatic relations was never credited, and AI diplomatic decisions (treaty proposals, war declarations) were generated but silently discarded. This was the single largest gap in the core game loop, affecting every game session.

b) Design Decisions

Direct Integration vs. Adapter Pattern

The existing diplomacy.ts module uses a separate DiplomacyState based on nested Maps (Map<string, Map<string, DiplomaticRelationFull>>), whilst the game state stores diplomacy as Empire.diplomacy: DiplomaticRelation[]. Rather than building complex adapter functions to convert between these two representations each tick, I chose to implement the tick logic directly on the Empire.diplomacy[] shape used by the game state. This avoids conversion overhead and keeps the game loop self-contained.

The existing evaluateTreatyProposal() function from diplomacy.ts IS reused for AI treaty evaluation, bridging the two representations only where needed.

Attitude Decay Constants

Decay uses the same formula as diplomacy.ts: 2% per tick with a 0.5 minimum shift. This ensures attitudes always drift toward neutrality but never oscillate. Very small values (below 0.5) snap to zero to prevent floating-point drift.

Trade Income Positioning

Trade income from diplomatic relations is applied during the diplomacy tick (Step 8), before AI decisions (Step 9). This means AIs can factor current diplomatic income into their decision-making. The income is separate from the existing trade route system (trade.ts) which handles physical trade routes between star systems.

AI Decision Execution

AI treaty proposals use the existing evaluateTreatyProposal() evaluator for AI-to-AI interactions, maintaining consistency with the diplomacy engine's personality-driven logic. War declarations check preconditions (not already at war, not allied) before executing.

Key Functions

// Diplomacy tick (Step 8) — processes each empire's relations
function stepDiplomacyTick(state: GameTickState, events: GameEvent[]): GameTickState
  - Attitude decay: 2% per tick, minimum 0.5 shift, snap to 0
  - Treaty expiry: removes treaties past duration, emits TreatyExpired events
  - Status recalculation: hostile(<-25) / neutral / friendly(>=25) / allied(>=60)
  - Trade income: 5 credits per route + 10 bonus per trade treaty

// AI treaty execution
function executeAIDiplomacy(state, empireId, decision, events): GameTickState
  - Validates preconditions (not at war, no duplicate treaties)
  - Uses evaluateTreatyProposal() for AI-to-AI acceptance logic
  - Adds treaty to both sides, applies attitude bonus

// AI war execution
function executeAIWar(state, empireId, decision, events): GameTickState
  - Validates preconditions (not already at war, not allied)
  - Sets at_war on both sides, clears treaties, applies -60 attitude penalty

f) Integration Notes

System Interactions

  • Economy: Trade income from diplomatic relations feeds into the credit pool before AI decisions, allowing AIs to factor diplomatic income into build priorities.
  • AI Engine: The evaluateDiplomaticActions() function in ai.ts now has its output executed rather than discarded. Personality weights for diplomacy and war decision types are now meaningful.
  • Happiness: The empireIsAtWar() check in happiness.ts now reflects dynamically-changing war states rather than only manually-set ones.
  • Victory: Diplomatic victory progress (alliance count) now changes dynamically as AI empires form and break alliances through the treaty system.
  • Espionage: Spy capture attitude penalties (from stepEspionage) now interact with the diplomacy tick's attitude decay, creating a natural recovery mechanic.
  • Combat: War declarations from the AI trigger at_war status, which the combat system uses for engagement rules.

Processing Order

The diplomacy tick runs at Step 8, after research (Step 7) and before AI decisions (Step 9). This means:

  1. Attitude decays and status updates apply before AIs make this tick's decisions
  2. AI diplomatic decisions from this tick take effect immediately for the next tick's processing
  3. Treaty expiry events are available for UI notification before the tick completes

g) Potential Improvements

  • AI Peace Negotiations: Currently, the AI can declare war but has no mechanism to propose peace. Adding peace proposals when attitude recovers above a threshold would complete the war → peace → treaty lifecycle.
  • Player Treaty UI: The ProposeTreatyAction, AcceptTreatyAction, and RejectTreatyAction types exist in events.ts but are not yet processed in the game loop's player action handler.
  • Diplomatic Incidents from Combat: When empires' fleets clash, this should generate attitude penalties. Currently, combat outcomes don't feed back into the diplomacy system.
  • Trust Tracking: The diplomacy.ts module has a full trust system (separate from attitude) that isn't reflected in the DiplomaticRelation type. Adding trust to the game state would enable deeper AI evaluation.
  • Treaty Counter-Proposals: The evaluator supports counter-terms but these aren't surfaced through the game loop.
  • Third-Party Attitude Effects: Declaring war on Empire B should negatively affect attitude with Empire C if B and C are allied.
  • Difficulty-Based AI Diplomacy: Higher difficulty AIs could use more sophisticated treaty evaluation, weighing long-term strategic value over immediate attitude scores.
